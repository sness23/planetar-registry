# planetar-registry — canonical data model: vertical slice

Working prototype of the canonical data-model spec
(`~/data/vaults/docs/ARCH-canonical-data-model.md`). Zero dependencies.

```
node demo.mjs          # one type end-to-end: registry -> codegen -> 3 encodings
node demo-fusion.mjs   # two types + identity resolution (dark-vessel re-id)
```

## demo.mjs — one type, every layer (8/8 pass)

`planetar:Vessel` taken from a real planetar-ais payload through:

1. **One registry entry → four artifacts** — JSON Schema, zmesg binary
   field-dictionary, TypeScript interface, SQLite DDL (see `gen/`). All are
   projections; none defines the model.
2. **Three encodings round-trip exactly** — JSON (warm), markdown+YAML (cold),
   binary-in-zmesg-frame (hot).
3. **Binary body 31% smaller than JSON** (219 B vs 316 B), numeric field-ids
   keep zmesg zero-copy intact.
4. **A namespaced third-party field (`acme:hull_temp_c`) round-trips** through
   all encodings and passes the core schema untouched.
5. **Governance is mechanical** — un-namespaced unknowns rejected by the
   generated schema's `additionalProperties:false`; namespaced keys pass via
   `patternProperties`. The regex *is* the rule.

## demo-fusion.mjs — identity resolution (5/5 pass)

A second type, `planetar:Detection` (a SAR radar blob with **no MMSI**), is
resolved against `planetar:Vessel` entities by the generic engine in
`resolve.mjs` — driven entirely by the registry's `resolution` config, no
maritime logic hardcoded. This is dark-vessel re-identification in miniature:

| Scenario | Input | Rule fired | Outcome |
|---|---|---|---|
| A | AIS re-observation (same MMSI) | `exact_id` → 1.00 | **MERGE** |
| B | SAR blob 0.3 km / 5 min from a vessel | `geo_time` → 0.50 | **REVIEW** (corroboration) |
| C | SAR blob, no AIS within 2 km | none | **NEW → dark-vessel candidate** |

The same engine, given an `exact_id`+fuzzy config, resolves a researcher to a
party — one resolver, many domains (spec §5.2).

## File map

| Path | Role |
|---|---|
| `registry/*.json` | git-vault registry SSOT — `planetar:Vessel`, `planetar:Detection`, namespaces, an `acme` extension |
| `src/registry.mjs` | loads the registry into the in-memory runtime model |
| `src/codegen.mjs` | registry entry → JSON Schema / field-dict / TS / SQL |
| `src/codecs.mjs` | JSON, markdown, binary, zmesg-frame codecs |
| `src/validate.mjs` | zero-dep stand-in for AJV (the governance check) |
| `src/resolve.mjs` | generic identity resolution, registry-driven |
| `src/adapter-ais.mjs` / `adapter-sar.mjs` | versioned source adapters → canonical entities |
| `sample/*.json` | real-shape planetar-ais and planetar-sat payloads |
| `demo.mjs` / `demo-fusion.mjs` | the two end-to-end runs |
| `gen/` | codegen output (regenerated by `demo.mjs`) |

## Findings (fed back into spec §13)

- **Kernel ≈ zmesg, confirmed 1:1.** No new envelope. Semver `schema_version`
  packs into zmesg's u32 field; canonical `type` is zmesg's `schema_name`.
- **Kernel-field collision is a real rule.** A source payload field colliding
  with a kernel name (`type`, `id`, …) must be renamed by its adapter —
  planetar-ais `type` → `vessel_class`, `length` → `length_m`.
- **Field-ids are type-scoped, not namespace-global.** The zmesg envelope names
  the type, so a `local_id` need only be unique within its type. `Vessel.mmsi`
  and `Detection.scene_id` can share `local_id` 1 without collision.
- **Identity resolution generalises cleanly.** One registry-driven engine spans
  ID-match (AIS) and spatial-temporal match (SAR); a non-match is not an error,
  it is the dark-vessel signal.
- **Provenance is deliberately not in the binary hot path** — it rides
  JSON/markdown and the canonical store.
- **Open questions made concrete** — confidence calibration (the SAR adapter
  normalises `snr/20`; AIS hardcodes `0.95` — not yet a common scale) and
  cross-namespace `ns_id` allocation still need a registry authority.
